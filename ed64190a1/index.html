What is a “system”?

First of all, it is very important to understand the meaning of the word “system”. It is introduced as follows in the dictionary.

  • ** System: An overall cohesive system that organizes many things and a series of operations **

Converting this into IT terminology,

  • ** System: A collection of many “data” and “programs” in which “a series of processes” are ordered **

In other words, it is an image like “Plastic model completed by assembling individual parts in an orderly manner”. For that purpose, three types of skills are required.

** ① Skills to think about parts required for development ** ② Skills for making individual parts ③ Skills to build up in an orderly manner

Other than that, it will be a blind spot** “① Skills for thinking about parts necessary for development” ** I wrote an article in “Requirements definition-Articles that can become people who can design systems”.

And “②How to make individual parts” has already been learned by anyone who has studied Python. It is sufficient to write a program that uses For statements, IF statements, functions, and classes.

After that, you will be able to develop the system if you acquire the skills for “③ Build in order”. In this article, we will proceed so that you can acquire the two skills required for that.

** (1) Understand “Overview of system development” ** (2) Understand the “role of individual parts” that make up the system

First, to clear (1), Let’s understand the development method (MTV model) ** adopted by Django.

What is Django’s “MTV model”?

Django uses a technique called MTV model.

  • The essence is exactly the same as the MVC model used in other languages such as Ruby.

Meaning of “MTV”

MTV was created by taking the initials of the three major elements that make up Django.

  • M: Model
  • T: Template
  • V: View

You don’t have to understand exactly, so let’s understand the rough role.

First, as a premise, the system is made up of three elements: “screen, processing, data” **. When expressed in sentences, it looks like this.

  • ** Add some “processing” to “data” and then display on “screen” **

Take, for example, a system that analyzes sales data. As shown in the figure, “screen, processing, and data” are allotted ** starting from the user operation.

If this is applied to the MTV model, the division of roles can be organized as follows.

  • View (Processing): ** Central control over the role **
  • Model (data): Role to control the database**
  • ** Templage (screen) **: Role to control ** screen (web page)

In other words, **By linking the three elements of “screen”, “processing” and “data” in an orderly manner, The system acts as a one aggregate.

  • Definition of “system” (repost)- System: A collection of “programs” in which many “data” and “series of processes” are ordered

With the above, I think we can see (1) Overall image of system development. *If you need other explanations, I’d be happy if you could make a request in the comment column.

Django file structure

First, in order to know the goal, organize the file structure (=part structure) of the system created with Django. Let’s understand the role played by each file while linking with the MTV model.

The one that is especially confusing is “** what is different about the following files? **”.

  • “app_config/views.py and app_folder/views.py”
  • “app_config/urls.py and app_folder/urls.py”

The difference can be arranged as follows, but at first I do not understand the merit of dividing.

  • app_config: manages the entire system
  • app_folder: Manages individual features in the system

First of all, the necessary skill is app_folder skill to develop a specific function, Just copy and paste the app_config, and let’s proceed ** assuming that a working system is created first**.

Now let’s look at the file structure.

List of files used by Django

When you proceed with development with Django, you will end up with such a file structure.

It’s okay if you don’t understand what it says at this point, Please refer to it when you get lost in the middle of programming “What are you doing now?".

  • manage.py: A collection of functions required for development management called “management commands”
  • app_config/: Folder that stores files related to global settings -views.py: [View] Write a program that controls the entire system -urls.py: Set the URL to use -settings.py: Manage various settings such as DB settings
  • app_folder/: Folder that stores the program to be implemented in the system -admin.py: Configuration file for admin screen -views.py: [View] Describe each function implemented in the system -urls.py: File that sets the URL to be used in the web page -models.py: [Model] Describe the DB definition -forms.py: File that sets the input form used in the Web page -migrations/: Folder where the DB definition change history remains
  • templates/: [Template] Folder for storing HTML files -base.html: HTML of “outer frame (sidebar etc.)” ** which is displayed commonly on all web pages -app_folder/ -top_page.html: “Inside” HTML displayed on each web page
  • static/: Folder for storing images, css files, javascript files -css/: Folder to store CSS files -js/: Folder for storing JS files -images/: Folder for storing image files

Again, these files (=parts) play their individual roles, By assembling them properly, the whole works as a “system”.

  • Definition of “system” (repost)- System: A collection of “programs” in which many “data” and “series of processes” are ordered

Create a view that controls the internal system

As I mentioned at the beginning, there are two types of views.

  • app_config/views.py: Control the whole thing
  • app_folder/views.py: Controls individual features**

First, create app_config/views.py from the simplest one. Create views.py in the app_config folder and describe the contents as follows.

from django.shortcuts import render, redirect, reverse
from django.views import View
  
class index(View):
    def get(self, request, *args, **kwargs):
        return redirect(reverse('app_folder:top_page'))
index = index.as_view()

This is a setting such as “Skip to the top page if there is access to the system”. It’s okay if you write the same thing every time.

Then open the app_folder/views.py that controls the specific features of the system and copy the following This writes the function “Display top_page.html when the top page is accessed”.

from django.shortcuts import render
from django.views import View
  
class SampleView(View):
def get(self, request, *args, **kwargs):
return render(request,'app_folder/top_page.html')
top_page = SampleView.as_view()

Creating a URL dispatcher

Here, set the URL required to access the system from the Web screen. You can freely decide the following https://, for example, http://localhost:8000/my_app/top_page. *There is a restriction that special characters such as “!” and “?” are not used, but if it is lowercase or underbar, there is no problem.

There are two types of files to change.

  • app_config/urls.py: System-wide URL design
  • app_folder/urls.py: URL design within individual apps

The relationship between the two affects the order in which the URL is constructed, http://localhost:8000/<URL setting for app_config/urls.py>/<URL setting for app_folder/urls.py> URLs are constructed in this order.

First is app_config/urls.py.

from django.contrib import admin
from django.urls import path, include
from django.conf import settings
from django.conf.urls.static import static
from .import views

#URL overall design
urlpatterns = [
    # URL to access the app "app_folder" created this time
    path('app_folder/', include('app_folder.urls')),
    # If you don't specify any URL (already set to access "app_folder" automatically in app_config/views.py)
    path('', views.index, name='index'),
    #URL to access the administration site
    path('admin/', admin.site.urls),
]

# Set URL for publishing media files
urlpatterns += static(settings.MEDIA_URL, document_root=settings.MEDIA_ROOT)

Next, create urls.py in the app_folder/ folder and describe it as follows.

Only the top page (top_page/) is prepared yet, so urlpatterns is one line, The URL to be registered will increase as various functions are added.

from django.urls import path
from .import views

app_name ='app_folder'
urlpatterns = [
    path('top_page/', views.top_page, name='top_page')
]

As a matter to understand, http://localhost:8000/<URL setting for app_config/urls.py>/<URL setting for app_folder/urls.py>

According to the URL setting rules, the URL to access the top page is http://localhost:8000/app_folder/top_page And so on. This is how the URL dispatcher works.

result = SampleDB.objects.filter(sample1=input_data)

return render(request,'app_folder/page02.html', context=context,)
