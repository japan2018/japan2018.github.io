What is API (merit of using API)

API is an abbreviation for “Application Programming Inter face”. The direct translation is as follows.

  • Application
  • Using Programming
  • Connect

*Application means “Web service” or “system”.

Going a little deeper, the biggest advantage that an API needs is “** Borrow useful functions created by others from your own web services**” I think this is all.

In other words, you no longer have to worry about developing from scratch. Which would you choose when developing an e-commerce site, for example?

  • Make your own credit payment function
  • Use credit payment API provided by the card company

In this way, the option of “don’t make it by yourself”** is the greatest merit of the API.

Understand the API

Now that you know the benefits of APIs, let’s understand how they work. Broadly speaking, three kinds of knowledge are required.

    1. Web structure
    1. HTTP request
    1. HTTP response

1) Web structure

The Internet, which I usually use casually, consists of the Web. The Web is a “mechanism to send all kinds of data such as sentences and images on the Internet”.

So, of course, you need to understand the web to borrow web services created by others using APIs. In order to understand the Web, let’s do the following three things.

  • HTTP (Hyper Text Transfer Protocol)
  • URL (exactly URI)
  • Hypertext (HTML etc.)

This is how it is organized in sentences.

What is Web? “According to the promise of HTTP communication” “To and from the location (server) specified in the URL” A mechanism to “exchange hypertext (information)”.

In order to understand the web in detail, I have to read many books, It is enough to understand that it is “technology for exchanging information with the server specified by the URL **”.

The API is a mechanism that allows you to easily exchange information between your own web service and **a web service (server) created by another person using this web technology.

To use the API specifically, it is necessary to use HTTP (communications convention) adopted by the Web. There are two types: “Send information to the other party HTTP request” and “Receive information from the other party HTTP response”, so let’s understand them later.

2) HTTP request

First, it is an HTTP request to send data from yourself to the other party. It is a process of sending a request such as “Please let the function be used” ** to the Web service created by another person via the API.

The API usage looks different for each Web service, but there is a common basis. The following information should be prepared in common when using any API.

  • Information for connecting to the Web service (available when registering the service) -API endpoint (URL to connect to the service) -API key (password required for connection)
  • Information to be sent to the web service (can be confirmed in API specifications) -Method (mainly POST or GET) -Query parameters (specify details such as the type of API and service to use) -Header (Enter authentication information such as data type and API key *Specify with POST method) -Body (enter the information you want to send and receive with the API. *Specify with the POST method)

These major items are common to all APIs, but there are differences in the details. Check the required contents in the “API specification” published by each service. *It is best to read the service usage procedure posted on Qiita until you get used to it.

If you use an API that can get the weather of each place, information will be organized in this way. You don’t have to be aware of it at first because it is written in the API specification as to whether to use POST or GET. *If you want to know the difference in detail, please refer to [here].

In addition to POST and GET, there are methods called PUT and DELETE, It is roughly described as follows.

  • GET: Get data
  • POST: Create new data
  • PUT: Update data
  • DELETE: Delete data

Weather forecast API (for POST method)

  • Information to connect to the web service -API endpoint: https:/world_weather.com/api *This is a fictitious URL -API key: aaabbbccc
  • Information sent to the web service -Method: POST -Query parameter: service=Weatyer (specify the service to use) -Header: application/json (data type) -Body: date=today, area=Tokyo (enter the information you want to send/receive)

The sample program is as follows. *Because the API does not exist, an error will occur even if it is executed.

#Import library
import requests

# Information to connect to the API
API_Endpoint = https:/world_weather.com/api
API_Key = aaabbbccc

# Information to send to API
headers = {'Content-Type':'application/json','key':API_Key}
body = {date='today', area=Tokyo}

# Make API connection
result = requests.post(API_Endpoint, data=json.dumps(body), headers=headers)

Weather forecast API (for GET method)

  • Information to connect to the web service -API endpoint: https:/world_weather.com/api -API key: aaabbbccc
  • Information sent to the web service -Method: GET -Query parameters: service=Weatyer, date=today, area=Tokyo (In the GET method, everything is stored in the query parameter)

The GET method depends on the service, but the following writing method is common.

** ① Principle: Include all information in the endpoint (URL) **

#Import library
import requests

# Information to connect to the API (connect the condition with "&" after "?")
API_Endpoint = https:/world_weather.com/api?Key=aaabbbccc&date='today'&area='Tokyo'

# Make API connection
result = requests.get(API_Endpoint)

② requests module function: Separate information like POST

The description method here is similar to POST. On the back side, the data will be sent in the same way as in “① Principle”, so the result will be the same.

#Import library
import requests

# Information to connect to the API
API_Endpoint = https:/world_weather.com/api
API_Key = aaabbbccc

# Information to send to API
headers = {'key':API_Key}
params = {date='today', area=Tokyo}

# Make API connection
result = requests.get(API_Endpoint, headers=headers, params=params)

3) HTTP response

When you send an HTTP request to a web service via the API, you will receive a “reply” called an “HTTP response”. By displaying this “reply” on your own Web service, you can save yourself the trouble of developing from scratch.

For example, [Request: Please tell us the weather. Today, Japan, Tokyo”

“Response: It’s sunny and cloudy”

API service example

Now that you have an overview of the API, let’s use a free API service that you can actually use.

Overview of API service used

Let’s use Livedoor weather information service Weather Hacks. You can get weather forecasts for 142 locations all over Japan for free up to today, tomorrow, and tomorrow.

http://weather.livedoor.com/weather_hacks/webservice

Take a look at the API specifications

Click the link above to see the API specifications.The number and names of items vary depending on the service, but the general order to check is as follows.

  1. What is the API endpoint?
  2. How to get API Key?
  3. POST or GET?
  4. What are the query parameters?
  5. What information should be sent in the header/body?

Let’s organize them in order. This time it’s a free service, so it’s very simple.

  1. **What is the API endpoint? ** http://weather.livedoor.com/forecast/webservice/json/v1 *End points may be described by URL/URI.

  2. **How to get API Key? ** Since there is no particular description, it seems to be a service that can be used without authentication (API Key not required).

  3. **POST or GET? ** This service uses the GET method. If the method is not specified even if you look at the specifications like this time, it is determined as follows. -If there is a description used in the form of “endpoint (URL) + query parameter”: GET is often found -Other than that: in most cases POST

  4. **What are the query parameters? ** It seems that you should specify the region (city) you want to get the weather forecast. Let’s use “Kurume: 400040” written in the sample.

  5. **What information should be sent in the header/body? ** Since GET method is used, it is not necessary to prepare header body.

Python code sample

Now that we know the information needed for the request, let’s make a request to the API in Python.

Submit API request

All you have to do is copy and paste the code and run it as is.

# Required libraries for API request
import requests
#URL + query parameter
url ='http://weather.livedoor.com/forecast/webservice/json/v1?city=400040'
# Submit API request
tenki_data = requests.get(url).json()

Displaying the returned results with print() can be very difficult to read. You cannot use it as it is, so you need to understand how to use JSON data.

print(tenki_data)

Confirm JSON data of API response

The data returned from the API is often in JSON format. Detailed explanation is omitted, but it is generally used after converting from JSON format to dictionary type in the .json() part included in the requests.get(url).json() of the code. It is target.

The API specification describes how to use the dictionary type.

http://weather.livedoor.com/weather_hacks/webservice

First, it says that you can get “title/heading” by specifying title in the dictionary type.

print('title:', tenki_data['title'])

Then, if you specify forecasts, it seems that you can get “weather forecast for each forecast day”. However, we need some complicated steps peculiar to the JSON format, so let’s hold down the flow together.

print(tenki_data['forecasts'])

First, when you specify forecasts, the list type with the first and last enclosed by [] is displayed. You can expect to specify “Today:[0], Tomorrow:[1], Tomorrow[2]”” according to the data you want to get.

# Specify "Today" data
print(tenki_data['forecasts'][0])

It was pretty refreshing. If you come to this point, you can collect your favorite data according to your purpose, so let’s display “Forecast Date: dataLabel, Weather: telop” this time.

# Forecast date
print('Forecast date:', tenki_data['forecasts'][0]['dateLabel'])
#Weather
print('weather:', tenki_data['forecasts'][0]['telop'])

In this way, a series of steps of “API specification check-request-response” flow. Various other information can be obtained, so please try variously referring to the API specifications.
