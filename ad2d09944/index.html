[Python] Why is the Python logging library camelCase?

Apr 7, 2020 Python coding standard PEP8 logging 

When I was talking about Python coding standards with a friend, I suddenly said, "Why some libraries ([`logging`](https://docs.python.org/ja/3/library/logging.html)etc.) does camelCase have been adopted?”

#logging example code. The method name such as getLogger is camelCase.
import logging

logger = logging.getLogger('simple_example')
logger.setLevel(logging.DEBUG)

In fact, PEP8 “Function and variable names” says:

Function names should be in lower case only. Also, for readability, words should be separated by underscores as needed.

The names of variables follow the same convention as functions.

Only allow mixedCase for compatibility if it is already in use (eg threading.py).

It says “I’m sorry that camelCase is already used in some libraries”. I was a little worried, so I tried to summarize the process. To conclude first,

  • Affected by Java library
  • Developed at the same time as PEP8 was established or earlier
  • Not modified for compatibility

That was the reason.

logging was developed at the same time as PEP8 was formulated with reference to Java

stackoverflow’s [How come the Python’s logging module doesn’t follow PEP8 conventions?”,IlearnedthatloggingwasoriginallyimplementedwithreferencetoJava’sLog4j.ItwasconfirmedthatPEP282 was listed as an “influenced library”.

Looking at the code for Log4j in TECHSCORE article, it is almost the same.

// create an instance
Logger logger=Logger.getInstance(Sample.class.getName());

// Set log event to occur when log level is WARN or higher
logger.setLevel(Level.WARN);

// log output
logger.info("This is info."); //No output
logger.warn("This is warn."); //output
logger.error("This is error."); //output

It also states the reason that “logging was already developed in 2001, at the same time that PEP8 was established.”

Also, according to the wiki, it is known among developers that it violates PEP8, but it is not consistent in the project or backwards. It seems that it has not been modified so far for compatibility.

API uses camelCase (goes against PEP8 recommendation and most of the stdlib)

  • PEP8 says-consistency with this style guide is important.Consistency within a project is more important.Consistency within one module or function is most important.
  • It’s a low priority right now, unless there’s an initiative to ensure the rest of the stdlib is made to conform to PEP8.– VinaySajip

About threading library

Also check PEP8’s threadingwhichwasmentionedasanexampleof"whenmixedCaseisalreadyused”.Itwas.Python3.8documentation is about “camelCase is adopted for backward compatibility”.

Although not listed here, camelCase names used in some methods and functions of this module in the Python 2.x series are still supported in this module.

In Python2.7, it was written like this.

As of Python 2.6, this module provides PEP 8 compliant aliases to replace camelCase properties affected by Java’s threading API. This updated API is compatible with that of the multiprocessing module. However, the camelCase name has not been deprecated and will continue to be supported in both Python 2.x and 3.x.

While maintaining compatibility, it is gradually migrating in a form that complies with PEP8. I would like the logging library to gradually move to a PEP8-compliant form as well as threading.

According to docs, this library seems to be strongly influenced by Java.

The rough design of this module is based on the Java threading model. However, while Java makes locks and condition variables the basic behavior of all objects, Python separates them into separate objects.
