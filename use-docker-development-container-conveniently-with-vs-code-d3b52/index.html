Use Docker development container conveniently with VS Code

Jan 17, 2020 Python Node.js development environment Docker VSCode 

#Introduction
  • Developing in local environment, deploying to Linux environment and testing is troublesome
  • I want to create a development environment that feels good to use Docker
  • However, various settings and construction are troublesome

For such a person, it is an introduction of how to use the extension function “Remote-Containers” that can conveniently build and operate a development environment using Docker container from VS Code.

Greatness of this extension

VS Code’s extension function “Remote-Containers” is to launch VS Code in a container and communicate with VS Code of the host machine so that you can perform in-container development as if you were developing in a local environment. is.

The detailed structure is shown in the official document.

(https://code.visualstudio.com/assets/docs/remote/containers/)

You can also manage multiple development environments from VS Code and launch a container with one click.

(https://code.visualstudio.com/assets/docs/remote/containers/)

Therefore, when you start development, you do not need to start the container from the command, attach the shell, and enter the container.

You can start developing in a container in the same way as opening VS Code in a local environment and starting development.

The system requirements are as follows.

-Windows: Docker Desktop 2.0+ on Windows 10 Pro/Enterprise.(Docker Toolbox is not supported.) -macOS: Docker Desktop 2.0+. -Linux: Docker CE/EE 18.06+ and Docker Compose 1.21+. (The Ubuntu snap package is not supported.)

Development environment construction

Project structure

Let’s say you want to create a Python application development environment.

Open the project directory from VS Code with the following directory structure.

project/
    └ .devcontainer/
        ├ Dockerfile
        └ devcontainer.json
    └ .git/
    └ package/
        ├ __init__.py
        ├ __main__.py
        └ module.py
    ├ requirements.txt
    └ .gitignore

Click the f1 menu or the green icon that appears in the bottom left and select “Remote-Containers: Open Folder in Container…”.

(https://code.visualstudio.com/assets/docs/remote/containers/)

Then VS Code reads the Dockerfile under .devcontainer and devcontainer.json, and launches the Docker container according to the settings.

Now let’s look inside the Dockerfile and devcontainer.json to understand what’s going on.

Dockerfile

This is an ordinary Dockerfile, and there is nothing special about it, but it does nice settings for user permissions.

FROM python:3

This item specifies the image that is the base for building the Docker image.

If you want to specify the python version in detail, enter python:3.7 here.

ARG USERNAME=vscode
ARG USER_UID=1000
ARG USER_GID=$USER_UID

RUN apt-get update \
    && apt-get -y install --no-install-recommends apt-utils dialog 2>&1 \

    && apt-get -y install git iproute2 procps lsb-release \

    ~~~

    && groupadd --gid $USER_GID $USERNAME \
    && useradd -s /bin/bash --uid $USER_UID --gid $USER_GID -m $USERNAME \

    && apt-get install -y sudo \
    && echo $USERNAME ALL=\(root\) NOPASSWD:ALL >/etc/sudoers.d/$USERNAME\
    && chmod 0440 /etc/sudoers.d/$USERNAME \

Here, apt-get settings and user permissions are set.

The point is that it seems that the container can be handled by a user named vscode who has the privilege to sudo.

devcontainer.json

Here is the miso of this extension.

"name": "Python 3",
"context": "..",
"dockerFile": "Dockerfile",

"settings": {
"terminal.integrated.shell.linux": "/bin/bash",
"python.pythonPath": "/usr/local/bin/python",
"python.linting.enabled": true,
"python.linting.pylintEnabled": true,
"python.linting.pylintPath": "/usr/local/bin/pylint"
},

"appPort": [9000],

"postCreateCommand": "sudo pip install -r requirements.txt",

"remoteUser": "vscode",

"extensions": [
"ms-python.python"
]
}

The items written in json format here are the settings for the Remote-Containers extension.

  • name -You can specify the display name of the Dev Container (different from the Docker container name).
  • context -Specify the project root. -In this case, “..” is specified because it is one level above the .devcontainer directory where devcontainer.json is located.
  • settings -It is an item to perform various settings in VS Code on the container side. Shell and Python paths etc.
  • appPort -Specify the port number of the container that can be accessed from the host machine. This is the same as specifying 9000:9000 with the -p option when running-docker run. -So hitting localhost:9000 in your host machine’s browser will connect to the container’s localhost:9000.
  • postCreateCommand -You can write commands that will be executed inside the container after building the image. -In this case, the package described in requirements.txt at the root of the project will be installed.
  • extensions -You can specify the extension installed in VS Code on the container side.

According to the official document, there are various other items that can be set.

By default the project root directory is bound to the container /workspace,

{"workspaceFolder": "/home/vscode",
    "workspaceMount": "type=bind,source=${localWorkspaceFolder},target=/home/vscode/project"
}

Will also be the default directory bound to /home/vscode and opened in VS Code.

{
    "containerEnv": {
        "WORKSPACE": "/home/vscode"
    }
}

If you specify the containerEnv item, you can set the environment variables that can be used in the container.

{
    "runArgs": [
"--name=project_dev_container"
    ]
}

You can also directly specify the option at container startup in the runArgs item.

Actually, VS Code loads this devcontainer.json, attaches various options, and runs docker to launch the container.

At that time, the list of character strings specified in the runArgs item is added separated by spaces.

More details here: Developing inside a Container-devcontainer.json reference

Other individual settings

git

If .git/ is included in the project directory that is bound to the container, the version control function of VS Code in the container can be used as it is.

The version control function of VS Code is very convenient, and you can perform git add, git commit, and git push with the GUI.

However, every time you try to communicate with a mote repository inside a container, you’ll need to authenticate to Github.

However, there is a function that allows us to share our VS Code and git credentials with the host machine.

Developing inside a Container-Sharing Git credentials with your container

First, save the Github user name and email address in the .gitconfig file with the host machine.

$ git config --global user.name "Your Name"
$ git config --global user.email "your.email@address"

These settings are written in .gitconfig in the user root, but VS Code seems to copy them to the container automatically.

Next, there are two ways to set authentication information such as passwords.

If you are authenticating with id and password using https communication, save the password in credential helper of git and that setting will be synchronized with the container.

Caching your GitHub password in Git

$ git config --global credential.helper wincred

With ssh authentication, if you register the public key for Github in SSH agent on the host machine, the settings seem to be synchronized.

With PowerShell

ssh-add $HOME/.ssh/github_rsa

Then the key will be registered in SSH agent.

However, the SSH agent is often not started, and at that time, enter PowerShell with administrator privileges.

Set-Service ssh-agent -StartupType Automatic
Start-Service ssh-agent
Get-Service ssh-agent

Alternatively, you can configure it from the GUI> Services> OpenSSH Authentication Agent Properties.

For details, see Using ssh-agent of Windows 10 with command prompt, WSL, and Git Bash, enabling #ssh-agent- etc.

AWS Access Key

When I try to communicate with AWS S3 from within the container, I get an access key issue.

I have access key information in the .aws directory in the user root of the host machine and I want to read it in the container as well.

However, unlike git, it doesn’t seem to load it automatically.

Therefore, you need to copy it to the container from outside the container using docker cp.

docker cp $HOME/.aws {container name}:home/vscode

Here, it is convenient if the container launched by Remote-Containers has a name.

In the runArgs item of devcontainer.json,

{
    "runArgs": [
"--name=project_dev_container"
    ]
}

It’s a good idea to give the container a name like this.

#Great voyage

When you actually use it, you can handle the development container with almost the same feeling as you normally develop with VS Code.

Also, when developing with Python, it is usual to prepare a virtual environment, but by constructing the environment inside the container, it is not necessary.

Because each project uses a different container, global installation of the package does not pollute the environment.

And since the image is prepared for each Python version, you can specify it according to the production environment.

(This is the advantage of using docker rather than Remote-Containers)

You can open and switch these development environments with a single click.

This extension also automatically binds ports, so you can use the local server for front-end development without stress.

However, for example, in node.js etc., the local server starts up by specifying port 3000, so that port must be published to the local machine.

In that case in devcontainer.json

{
    "appPort": [3000]
}
