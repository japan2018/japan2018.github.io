Constructor

I think that you previously added the data attributes later. However, that does not bring out the goodness of object orientation. This section describes the Constructor that adds data attributes when instantiating.

A constructor is a process that is executed once when it is instantiated. Simply put, it is Initialization.

Correction: The __init__ method is definitely an initialization method, but there is another __new__ method in the constructor, so the “constructor=__init__ method” seems to be wrong. It is possible to call it later. ,


class Person:
  def __init__(self,name,age):# This is the constructor
    self.name = name
    self.age = age

person = Person("mackerel", 19) # There is an argument when instantiating.

A statement described as __init__ can be executed with an argument only once when instantiated. At this point, you may feel that “Is it okay to substitute it later?”, but if you use it with the encapsulation described later, the effect can be brought out, so please listen to it now.

Encapsulation

I first mentioned that python has a loose language specification, which allows you to later access the attributes of a class and assign negative or different values.

When defining a python class,

Enter the name of the attribute you want to restrict access to Start with an underscore (_)

It seems that the method is practiced as a custom.


class Person():
  def__init__(self,name,age):
    self._name = name # <-with an underscore
    ...

Even so, I think that you may mistakenly write the *** underscore (_)*** and rewrite it. You can force python to prevent rewriting. for that purpose

Make the attribute name start with two underscores (__)

Will be described


class Person():
  def__init__(self,name,age):
    self.__name = name # <-with two underscores
    ...

In this way, making the inside of the class invisible as if the attributes inside the class are encapsulated (concealing) is called encapsulation.

Addendum: It seems that encapsulation is not just about hiding, but encapsulation is about hiding complicated processes and protocols to make them easier to use. More generally, it seems that the double underscore method should only be used when there is a name conflict with a subclass. It’s not a recommended method. .. #Summary
