Necessity of requirement definition

Purpose of requirement definition

There are roughly two types of parties involved in system development projects.

  • ** Those who “make” the system (developers) **
  • ** People who “use” the system (orderers) **

If you want to create the system you want, there is a complete image in your head, so no problem, In many cases, we ask the “orderer ⇒ developer” to develop.

At that time, the developer does not know what to make without ** clarifying the image in the mind of the orderer. There is a requirement definition for the purpose of communication.

Let’s compare it to everyday life. When you’re thirsty, would you ask me to buy a drink?

You can say, “Get some coffee,” or more specifically “Bought some hot sugar coffee. It’s a plastic bottle, not a can!” I can’t get you the coffee I thought I wouldn’t say.

System development is the same. **“What would you like to make? The purpose of the requirements definition is to clearly agree on **.

That is, **Requirement definition = Advance promise that you can confirm concretely “This is OK!” at the time of delivery ** It is good to have an understanding such as.

Process for determining requirements definition

Now that we know the purpose of requirement definition, let’s understand the three stages until the requirement definition is completed.

  • Request: Idea such as “I wish there was such a system”
  • Request: I want you to implement it in the system Rough function list
  • Requirements: Both parties agreed Specific function list and Implementation method

Communication through “examination” and “suggestions”** is required for this three-step process. It is as follows when arranged in order.

  • ** ① Request: Issues to be solved (Orderer task) **
    • Current issues -Goal (what it should be) -Gap between current situation and goals (issues to be solved)
  • ** ② Request: Function to be implemented in the system (Orderer task) ** -Planning background (issues to be solved) -Overview of the system required to solve the problem -List of functions you want to implement concretely
  • ** ③ Examination: Considering the feasibility of requirements (developer task) ** -Is it technically feasible? -How much budget do you need? -When is the delivery date?
  • ** ④ Proposal: Return the result of consideration to the orderer (developer task) ** -Functions that can be implemented -Amount to charge -Delivery date
  • ** ⑤ Requirement: Decisions agreed by both parties (Orderer/Developer discuss and decide) ** -List of functions implemented in the system -In some cases, the estimated delivery date and billing amount are listed.

Matters determined by requirement definition

Now that we understand the flow, we will outline the items to be decided in the requirement definition. If you organize by 3W (Why, What, How), it will be neatly organized.

  • Why: Purpose of system development (request)
    • Current issues -Goal (what it should be) -Gap between current situation and goals (issues to be solved)
  • **What: How to solve the problem ** -Business flow after system introduction -Functional requirements -List of functions implemented in the system
    • Non-functional requirements -Various processing speed, security, etc.
  • How: Specific usability and implementation method (task similar to system design)
    • basic design -Screen design (UI design) -Functional design -Data design
    • Detailed design -Class diagram, sequence diagram -System architecture -Various technologies such as technology for mounting each part

How to make a basic design (screen design, functional design, data design)

You can imagine that there are so many items to be decided in the requirement definition, but please be assured.

What (work flow and functional/non-functional requirements) is not an engineer, Among the development side, there is a tendency that “** sales person **” and “** sales engineer **” are the corresponding jobs.

So, I will continue to dig deeper on how ** (engineering) engineers can demonstrate their abilities. Of these, in particular the basic design is key, so this article will focus on that.

As mentioned above, the basic design is to design the “screen, function, data” **. Only then can a programmer embark on software development.

  • ** Screen design (UI design) ** -What can you do on each screen? -Information to display (text, images, etc.) and layout -Summarize the above into a screen transition diagram
  • ** Functional design ** -Back side processing (function name and processing content) -Data required for processing, data acquisition source (input from screen, acquisition from DB, etc.) -Delivery destination of processed data (screen display, save to DB, etc.)
  • ** Data design ** -Concrete contents of data -Database design -Data flow

Screen design (UI design)

First of all, the element that should be decided before anything else is screen design (UI design).

Since the user uses the software through the screen (UI), First of all, getting the consent to “Ease of use of screen”** is the shortcut to the success of the project.

As mentioned above, the following items are decided in the screen design.

  • What can you do on each screen?
  • Information to display (text, images, etc.) and layout
  • Summarize the above into a screen transition diagram

By organizing up to this point, you can get a concrete image of the completed system. This is the reason why the screen design is made first.

Functional design

After the screen design (UI design) is finished, design the functions to be implemented in the system. Please understand that the functional design is the phase in which “the processing to be executed on the other side, necessary data” ** is defined.

The necessary decisions are as described above.

  • Back side processing (function name and processing content)
  • Data required for processing, data acquisition source (input from screen, acquisition from DB, etc.)
  • Delivery destination of processed data (screen display, save to DB, etc.)

In this way, anyone can see what kind of program should be written. Since the parts that make up the system are clarified, the division of roles within the team is also easy.

Data design

The last is data design. With the preparations made so far, the data design can finally begin.

As mentioned above, there are three items to be decided.

  • Concrete contents of data
  • Database design
  • Data flow

The point to be aware of here is that there are 4 types of ** data. Design how these four types of data flow in the system.

  • Data to be input to the program (argument) -Data entered by the user from the web screen -The data to read from the database
  • ** “Return” data from the program (return value) ** -Data to be displayed on the web screen -Data stored in database

Specific contents of data

At this stage, the abstracted data definitions such as “Member information” are embodied. At the practical level, the knowledge of “normalization”** is required. *If you are interested, please see [Reference Article].

  • Member Information
    • Full name -Phonetic
    • mail address -Password
  • ** Product information **
    • Product name -Detailed description -Product image URL -Genre
    • Maker -Price -Rating
  • Purchase history -Member ID -Product ID
  • Purchase date -Purchase quantity

Database design

After clarifying the contents of the data, design the database.

The first thing to remember is that until now, data was called by “member information”, “product information”, “purchase history”, etc. From a database perspective, we use the term table. *** “Member table,” “Product table,” “Purchase history table,” etc.

There are three tasks required in database design.

  • Divide table role into “master” and “history”
  • Organize reference relationships between tables
  • Transcribe the reference relationship into an ER diagram

*(Note) The names such as “master” and “history” are from my own point of view.

First, let’s divide the role of each table according to the following criteria.

  • ** Master: Basic information (data referenced from other tables) ** -Member table (hereinafter referred to as “member master”) -Product table (hereinafter referred to as “Product master”
  • ** History: Data accumulated as the system is used ** -Purchase history table

Next, “reference relationship between tables” organizes the relationship of reference destination/reference source with arrows. It is easy to understand if there is no omission of data flow if Data that results from user behavior is also specified. *In some cases, data such as the weather forecast API is brought in from outside via a network.

Finally, create an ER diagram, which will be the blueprint for the database, and it is complete. *Please refer to [Reference Article] for details on how to write an ER diagram.
